# 知识库与龙头映射：实施开发说明

本文档在 [知识库与龙头映射设计.md](./知识库与龙头映射设计.md) 的产品与数据形状约定之上，按**当前仓库代码**拆解为可执行的开发任务、文件清单、接口约定与验收标准。实现完成后，流水线应在 `relevance_filter` 之后用**静态配置表**填充 `market_map` / `market_impact`，供 `node_card_renderer` 与飞书展示使用。

---

## 1. 目标与不做的事情

### 1.1 目标（本迭代交付）

1. 根据 `analysis.themes` / `analysis.keywords`（及可选正文）从**版本化静态表**解析出 `market_impact`（`direction`、`sectors[]`、`stocks[]`、`mapping_confidence`）与 `market_map`（命中过程元数据）。
2. 在 LangGraph 中**插入** `market_retriever` 节点：仅在 `relevance_filter` 放行分支执行，位于 **`body_translate` 之前**（与设计一致：映射只依赖 `analysis`，不依赖译文；译文明文仍可选参与 keyword 子串匹配）。
3. 通过 `config.toml`（及可选环境变量）控制**是否启用**、**表文件路径**，默认关闭或开启时路径无效应**可观测失败**（日志 + 安全降级策略见下文）。

### 1.2 非目标（可排期后续）

- 向量检索、在线拉取股票基础信息 API。
- 二次 LLM 在候选内消歧（设计文档 §7 可选扩展）。
- 修改 `triage_system.txt` 的 theme 枚举（若改枚举，必须同步改 YAML；本任务书默认**沿用现有枚举**）。

### 1.3 与现有代码的契约

| 字段 / 行为 | 现有约定 | 实现时注意 |
|-------------|----------|------------|
| `analysis.themes` | `triage_system.txt` 要求取自 `[semiconductor, ai, optical_module, macro, geopolitical, other]` | 表中 `theme` 键必须**完全一致**（大小写敏感）。 |
| `relevance_filter` 出口 | 已写入占位 `market_impact`（`_EMPTY_MARKET_IMPACT`） | `market_retriever` 应**覆盖**为规则表结果（或合并策略在设计阶段定死：建议**完全覆盖**）。 |
| `node_card_renderer` | 已读 `mi.get('sectors')` / `stocks` 等 | 保持 `market_impact` 形状与现有一致即可；若优化展示（如分行、中文名），在 renderer 增量改。 |
| `PipelineState` | 已有 `market_map`、`market_impact` | 一般无需改 `TypedDict`，除非增加调试字段（建议放进 `market_map`）。 |

```
knowledge/
  __init__.py           # 可空或导出 load_theme_leaders / match_market
```

## 4. 配置层开发任务

### 4.1 TOML 形状（建议）

在 `config.example.toml` 增加（默认值与文档一致）：

```toml
[knowledge]
enabled = false
theme_leaders_path = "knowledge/theme_leaders.yaml"
```

说明：

- `theme_leaders_path`：相对**进程当前工作目录**（一般为项目根）或绝对路径；与 `load_config` 的 `cfg_path` 无关，需在 `loaders` 内 `Path(path).expanduser()`。
- 若 `enabled = false`：`PipelineCompileConfig` 不传 matcher，图中**不插入** `market_retriever` 或插入 **no-op**（推荐不插入，减少一次节点调用）。

### 4.2 `ingestion/config.py`

1. 新增 `@dataclass(frozen=True) class KnowledgeConfig`：`enabled: bool`、`theme_leaders_path: str`。
2. 在 `AppConfig` 中增加字段 `knowledge: KnowledgeConfig`。
3. 在 `load_config` 中解析 `[knowledge]` 段；缺省时 `enabled=False`、`theme_leaders_path="knowledge/theme_leaders.yaml"`（或空字符串并在 enabled 时校验）。

### 4.3 `pipeline/graph.py` — `PipelineCompileConfig`

1. 增加可选字段，例如：`knowledge: Optional[KnowledgeConfig] = None`，以及 **`theme_leaders_table: Optional[ThemeLeadersTable] = None`**（类型在 `knowledge/loaders.py` 定义），或传入可调用对象 `market_match_fn`（便于测试注入）。
2. `from_app`：把 `app.knowledge` 拷入；若 `enabled` 且路径可读，在 `from_app` **或** 首次 `compile` 时调用 `load_theme_leaders(path)`，失败时策略二选一（**须在 PR 中明确**）：
   - **严格**：构图时 `raise`，防止静默无映射上线；
   - **宽松**：构图成功但 matcher 为 None，图中跳过 `market_retriever` 并打 warning（适合本地开发）。

推荐本地开发用宽松，CI 用严格。

## 5. 数据层：`theme_leaders.yaml`

### 5.1 逻辑模型（与设计文档 §4.2 对齐）

每条 row：

- `theme`：与 LLM 输出一致，取自 `triage_system.txt` 枚举。
- `sector`：`{ name, code }`（指数/行业体系全仓统一，如申万）。
- `leaders`：`[{ code, name, role }]`，`role` 建议 `leader` / `follower` / `other`。
- `keyword_boost`（可选）：子串列表，用于提升命中或并入该行。
- `priority`（可选）：整数，多 theme 合并时排序。

文件级：

```yaml
version: "2026-05-12"
rows:
  - theme: optical_module
    sector: { name: "通信设备", code: "SW801102" }
    keyword_boost: ["CPO", "光模块", "800G"]
    priority: 10
    leaders:
      - { code: "EXAMPLE.SH", name: "示例", role: leader }
```

**合规**：示例代码勿用未授权真实荐股名单；生产数据由业务在仓库外或私有分支维护。

### 5.2 `knowledge/loaders.py` 职责

1. `load_theme_leaders(path: Path) -> ThemeLeadersTable`：读文件、解析、校验必填字段；异常抛出带路径信息。
2. 内存结构建议：`dict[str, list[Row]]`（theme → rows）及 `version: str`，便于 `match`  O(1) 查 theme。
3. 启动策略：**进程内缓存**：首次构图或首次 `invoke` 时加载一次即可（注意测试里多次改文件需 reload 钩子，MVP 可忽略）。

## 6. 匹配层：`knowledge/match.py`

### 6.1 函数签名（建议）

```python
def match_market(
    analysis: dict[str, Any],
    *,
    table: ThemeLeadersTable,
    raw_text: str = "",
    raw_text_zh: str = "",
) -> tuple[dict[str, Any], dict[str, Any]]:
    """返回 (market_impact, market_map)。"""
```

### 6.2 算法（与设计 §5 一致，落地细节）

1. **Theme 直查**：对 `analysis.get("themes") or []` 中每个 `t`，取表中 `theme == t` 的所有 row，收集 `sector`、`leaders`。
2. **Keyword 增强**：将 `keywords` 与（可选）`raw_text`、`raw_text_zh` 拼成规范化字符串（如 `lower()`，注意中英文）；对每 row 的 `keyword_boost` 做**子串包含**；命中则并入集合（若尚未有该 theme 行）。
3. **去重**：`sectors` 按 `code`；`stocks` 按 `code`；`role` 冲突时取更高优先级（`leader` > `follower` > `other`）。
4. **`mapping_confidence`**：
   - 至少一次 **非 `other` 的 theme 直查** → `high`；
   - 仅 **keyword_boost** 命中 → `medium`；
   - 无命中 → `low`，`sectors`/`stocks` 为空列表。
5. **`direction`**：由 `analysis.sentiment.label` 映射：`positive`→`bullish`，`negative`→`bearish`，`neutral`→`neutral`；缺省或未知 → `neutral`。

### 6.3 `market_map` 建议形状（与占位节点兼容）

```python
{
    "source": "theme_keyword_table",
    "version": table.version,
    "matched_themes": [...],
    "matched_keyword_rules": [...],  # 命中的 boost 词或 row id
    "candidates": [],  # MVP 可空；预留二次 LLM
}
```

## 7. 节点层：替换 `node_market_retriever`

### 7.1 工厂函数（与 `make_relevance_filter` 同模式）

在 `pipeline/nodes.py` 中：

1. 保留或删除裸函数 `node_market_retriever`，改为：

   `def make_market_retriever(*, table: ThemeLeadersTable | None) -> Callable[[PipelineState], dict[str, Any]]:`

2. 若 `table is None`：内层返回 `{}`（不修改 state）或返回当前 `market_impact` 不变（与「图中不注册节点」二选一，**不要**既注册又 no-op 除非行为与文档一致）。

3. 若 `table` 非空：
   - 读取 `state["analysis"]`、`state.get("raw_text")`、`state.get("raw_text_zh")`；
   - 调用 `match_market(...)`；
   - 返回 `{"market_impact": ..., "market_map": ..., "status": "mapped"}`。

4. **不要**清空 triage 已写的其它 `analysis` 字段；只写 `market_*`。

### 7.2 `relevance_filter` 是否仍写空 `market_impact`

当前 `make_relevance_filter` 在返回前总是带 `_EMPTY_MARKET_IMPACT`。可选优化：

- **最小改动**：保持现状，`market_retriever` 覆盖即可。
- **更清晰**：relevance 阶段不写 `market_impact`，仅由 retriever 写入（需确认下游无依赖「必有键」；`card_renderer` 已用 `or {}`，可改）。

## 8. 图编译层：`pipeline/graph.py`

### 8.1 注册节点

在 `_register_nodes` 中：

```python
g.add_node(
    "market_retriever",
    nodes.make_market_retriever(table=self._cfg.theme_leaders_table),
)
```

（具体字段名以你实现的 `PipelineCompileConfig` 为准。）

### 8.2 改边（核心）

在 `_wire_entry_and_relevance` 中，将：

- `after_relevance` 的 `"continue": "body_translate"`  
  改为 **`"continue": "market_retriever"`**（当 knowledge 启用且表加载成功时）。

然后：

- `g.add_edge("market_retriever", "body_translate")`。

当 knowledge 未启用时：保持现有 `continue → body_translate`，**不**注册 `market_retriever`（推荐），避免无意义空节点。

### 8.3 条件边

`_EdgeRouter.after_relevance` 仍只看 `status == "filtered"`；`market_retriever` 返回 `status: mapped` 不影响 continue，无需新路由。

### 8.4 `build_pipeline_graph_from_app`

在调用 `PipelineCompileConfig.from_app` 时传入已加载的 `ThemeLeadersTable` 或 `None`，逻辑集中在 `from_app` 或 `TweetPipelineCompiler.__init__`。

## 10. 测试计划

### 10.1 单元测试 `tests/test_knowledge_match.py`

| 用例 | 输入 | 期望 |
|------|------|------|
| theme 直查 | `analysis={"themes":["semiconductor"],...}` + 含该 theme 的表 | `mapping_confidence == "high"`，sectors/stocks 非空 |
| keyword 仅命中 | themes `["other"]` + 正文含 boost 词 | `medium` 或按设计 |
| 无命中 | 空 themes、无关键词 | `low`，空列表 |
| direction | sentiment positive / negative | bullish / bearish |

使用**固定内存表**或临时 `tmp_path` 写 YAML，不依赖外网。

### 10.2 图集成测试 `tests/test_pipeline_graph.py`

1. 增加 `theme_leaders.yaml` fixture（放 `tests/fixtures/` 为宜）。
2. 构造 `PipelineCompileConfig` + 已加载表，`invoke_for_tweet` 后断言：
   - `out["market_impact"]["mapping_confidence"] == "high"`（对构造的 triage analysis，若测试中 mock LLM 困难，可暂时只测 **带 stub 的图**：见下条）。
3. **Mock LLM**：若 CI 无 API Key，现有测试依赖 stub；对 mapping 的集成测试可：
   - 仅测 `match_market` + `make_market_retriever` 单独调用；或
   - 使用 `PipelineCompileConfig(llm=None)` 时 stub analysis 里手工注入 `themes`/`keywords`（需在 `make_relevance_filter` stub 分支扩展测试钩子，**非必须**：单元测试覆盖 matcher 即可满足 MVP）。

### 10.3 手工验收

1. 配置真实 `theme_leaders.yaml`（小表）+ 打开 `[knowledge] enabled`。
2. 跑 `python main.py` 或单条脚本 `invoke_for_tweet`，检查飞书或 stdout 中板块/龙头与表一致、无臆造代码。
