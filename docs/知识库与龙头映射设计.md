# 知识库与龙头映射：详细设计（对接说明）

> **范围**：在 LLM triage 已产出 `analysis.themes` / `keywords` 之后，如何用**规则 + 静态配置表**（**不做向量检索**）填充 `market_impact`（板块、龙头标的），并接入现有 LangGraph。  
> **非目标**：本节不写具体股票名单维护流程、合规结论；不替代投研与交易决策。

---

## 1. 背景与目标

### 1.1 现状（仓库内）

| 组件 | 现状 |
|------|------|
| `relevance_filter` | DeepSeek 产出 `analysis`（含 `themes`、`keywords`、`sentiment` 等） |
| `market_impact` | 占位：`direction`、`sectors`、`stocks`、`mapping_confidence`（见 `pipeline/nodes.py` 中 `_EMPTY_MARKET_IMPACT`） |
| `node_market_retriever` | 仅占位：写空 `market_map.candidates`，**未查表** |
| `pipeline/graph.py` | 边为 `relevance_filter → body_translate → card_renderer`，**未插入** `market_retriever` |
| `node_card_renderer` | 已读取 `market_impact` 拼飞书文本 |

### 1.2 目标

1. **确定性**：龙头与板块代码来自**可版本化配置**，禁止仅依赖 LLM 随口生成代码。  
2. **可解释**：每条推送能说明「因 theme=X、keyword 命中 Y 而附带板块 Z、龙头列表 W」。  
3. **低延迟**：单条请求内完成检索，**无需向量库**。  
4. **可演进**：预留「候选集 + 二次 LLM 在候选内选择」的扩展位（可选）。

---

## 2. 总体数据流（对接后）

```
采集 TweetEvent
  → relevance_filter（LLM：themes + keywords + …）
  → market_retriever（本设计：读知识表 → 写 market_impact / market_map）
  → body_translate
  → card_renderer（已消费 market_impact）
  → feishu_publisher
```

与主设计文档 [产品愿景与设计边界.md](./产品愿景与设计边界.md) §4.4 一致：**LLM 做归类与抽取，代码做映射与约束**。

---

## 3. State 字段约定

与 `pipeline/state.py` 对齐：

| 字段 | 写入节点 | 含义 |
|------|----------|------|
| `analysis` | `relevance_filter` | 含 `themes: string[]`、`keywords: string[]`、`sentiment`、`confidence` 等 |
| `market_map` | `market_retriever`（建议） | **检索过程元数据**：命中规则、候选列表、调试信息（可选） |
| `market_impact` | `market_retriever`（建议） | **下游展示与产品语义**：方向、板块、标的、映射置信度 |

### 3.1 `market_impact` 建议形状（与卡片一致）

与当前 `node_card_renderer` 使用方式兼容：

```json
{
  "direction": "bullish | bearish | neutral",
  "sectors": [
    { "name": "半导体", "code": "SW801081" }
  ],
  "stocks": [
    { "code": "603986.SH", "name": "兆易创新", "role": "leader" }
  ],
  "mapping_confidence": "low | medium | high"
}
```

- `direction`：可与 `analysis.sentiment.label` 简单映射（如 positive→bullish），或由单独规则表覆盖；MVP 可直接沿用 sentiment。  
- `role`：建议保留 `leader` / `follower` / `other`，与主设计 §5 示例一致。

### 3.2 `market_map` 建议形状（可观测性）

```json
{
  "source": "theme_keyword_table",
  "version": "2026-04-12",
  "matched_themes": ["semiconductor", "optical_module"],
  "matched_keyword_rules": ["CPO", "光模块"],
  "candidates": []
}
```

`candidates` 预留给后续「二次 LLM」：每项为 `{ "theme", "sector", "stocks_preview" }` 等。

## 4. 知识库：静态表设计（无向量）

### 4.1 存放位置与格式

建议：

- 路径：`knowledge/theme_leaders.yaml`（或 `knowledge/theme_leaders.json`）。  
- **随 Git 版本管理**；CI 可校验 JSON Schema / 必填字段。  
- **不把**大段投研文本放进仓库时，可拆 `knowledge/sectors/*.yaml` 按 theme 分文件。

### 4.2 逻辑模型

**主题行（Theme row）**：与 triage prompt 中 `analysis.themes` 枚举 **字符串完全一致**（如 `semiconductor`、`ai`、`optical_module`、`macro`、`geopolitical`、`other`）。

每条记录建议包含：

| 字段 | 类型 | 说明 |
|------|------|------|
| `theme` | string | 与 LLM 输出一致 |
| `sector` | object | `name`、`code`（与你们采用的指数/行业体系一致，全仓统一一种标准） |
| `leaders` | array | `{ code, name, role }`，**人工或离线脚本维护** |
| `keyword_boost` | array（可选） | 当 `keywords` 或正文含某子串时，**提高**本 theme 命中权重或强制并入 |
| `priority` | int（可选） | 多 theme 合并时排序 |

`theme == other` 时通常 **不配置龙头**，或仅配泛化「宽基」占位（产品决定）。

### 4.3 示例（片段，非真实投资建议）

```yaml
# knowledge/theme_leaders.yaml（示例结构）
version: "2026-04-12"
rows:
  - theme: optical_module
    sector: { name: "通信设备", code: "SW801102" }
    keyword_boost: ["CPO", "光模块", "硅光", "800G"]
    leaders:
      - { code: "EXAMPLE.SH", name: "示例公司A", role: leader }
  - theme: semiconductor
    sector: { name: "半导体", code: "SW801081" }
    keyword_boost: ["晶圆", "先进制程", "EDA"]
    leaders:
      - { code: "EXAMPLE2.SH", name: "示例公司B", role: leader }
```

## 5. 匹配算法（MVP）

**输入**：`analysis.themes`、`analysis.keywords`、（可选）`raw_text` 或 `raw_text_zh` 拼接串。  
**输出**：合并后的 `sectors[]`、`stocks[]`、`mapping_confidence`、`market_map`。

### 5.1 步骤

1. **Theme 直查**  
   对每个 `themes` 中的 `t`，在静态表中取 `theme == t` 的行；收集 `sector`、`leaders`。

2. **Keyword 增强**  
   将 `keywords` 与（可选）正文转为小写或可配置规范化后，对每行的 `keyword_boost` 做**子串包含**匹配；命中则把该行并入结果集（若尚未存在）。

3. **去重**  
   - `sectors`：按 `code` 去重。  
   - `stocks`：按 `code` 去重；同一标的多 theme 命中可保留一条，`role` 取最高优先级（如 leader > follower）。

4. **mapping_confidence**  
   - 至少命中 **一个非 other 的 theme 直查** → `high`。  
   - 仅 **keyword_boost** 命中 → `medium`。  
   - 无命中 → `low`，`sectors`/`stocks` 为空数组（与当前占位行为一致）。

5. **direction**  
   MVP：`analysis.sentiment.label` 映射为 `bullish`/`bearish`/`neutral`；`mixed` 可映射为 `neutral` 或单独规则。

### 5.2 为何不需要向量检索

Theme 已是离散标签；龙头表是**精确映射**。向量适合「找不到 theme、只有长文本语义」的召回，本流水线在 triage 阶段已压缩为 **theme + keywords**，用规则即可稳定、可审计。

## 6. 代码对接清单（实现顺序）

### 6.1 新增模块（建议）

| 模块 | 职责 |
|------|------|
| `knowledge/loaders.py` | 启动时或首次调用时加载 YAML/JSON，缓存为内存结构 |
| `knowledge/match.py` | `match_market(analysis, raw_text?) -> (market_impact, market_map)` |
| （可选）`knowledge/schema.json` | 配置校验 |

### 6.2 修改 `node_market_retriever`

- 调用 `match_market(...)`。  
- 返回 `{"market_impact": ..., "market_map": ..., "status": "mapped"}`。  
- **不要**清空已有 `market_impact` 中将来可能由 LLM 写入的字段（若以后二阶段 LLM 先写草稿）；当前以规则结果**覆盖**占位即可。

### 6.3 修改 `pipeline/graph.py`

1. `_register_nodes`：`g.add_node("market_retriever", nodes.node_market_retriever)`（或 `make_market_retriever(knowledge_cfg)` 工厂注入配置）。  
2. `_wire_entry_and_relevance`：  
   - `relevance_filter` 的 `continue` 目标由 `body_translate` 改为 **`market_retriever`**。  
   - `g.add_edge("market_retriever", "body_translate")`。  
3. 若 `after_relevance` 需区分 `filtered` vs `continue`，保持不变；`market_retriever` 仅在 continue 分支到达。

### 6.4 配置（建议）

在 `config.toml` 增加可选段，例如：

```toml
[knowledge]
theme_leaders_path = "knowledge/theme_leaders.yaml"
enabled = true
```

`load_config` → `AppConfig` → `PipelineCompileConfig` → `make_market_retriever` 闭包读取路径与开关。

### 6.5 `setup.py` / 打包

若 YAML 随仓库发布，增加：

```python
package_data={"knowledge": ["*.yaml"]}
```

或保持 `knowledge/` 在项目根目录、通过 `path` 配置指向，**不必**打进 wheel（运维常用）。

## 7. 可选扩展：二次 LLM（候选内选择）

**触发条件**：`mapping_confidence == "medium"` 或命中多 theme 需消歧。

**流程**：

1. `market_retriever` 规则阶段产出 `market_map.candidates`（每 theme 最多 K 个板块 + M 个龙头）。  
2. 新节点 `market_disambiguator` 或使用同一节点内二次调用：prompt 要求 **仅从 candidates 中选**，输出 JSON。  
3. 合并进 `market_impact`，`mapping_confidence` 上调为 `high` 或标记 `reviewed_by_llm: true`。

本节为扩展，**MVP 可不做**。

## 8. 测试与验收

| 类型 | 内容 |
|------|------|
| 单元测试 | 给定 `analysis` + 固定 YAML，`match_market` 输出确定的 sectors/stocks |
| 回归 | 图编译后 `invoke`，断言 `market_impact.stocks` 非空（对构造的 theme） |
| 人工 | 飞书消息中板块/龙头与配置表一致，无臆造代码 |

---

## 9. 风险与约束

1. **龙头名单时效**：退市、更名、代码变更需更新表；可约定每周/每月同步脚本。  
2. **theme 漂移**：若 triage prompt 修改 theme 枚举，必须同步改知识表 `theme` 键。  
3. **合规**：自动推送需免责声明（已有）；龙头列表不代表推荐买入。  
4. **多数据源**：CLS / X 共用同一 pipeline，`market_retriever` **与 source 无关**，仅依赖 `analysis`。
