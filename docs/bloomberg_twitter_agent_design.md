# Bloomberg Twitter Agent：A 股影响监控 — 详细设计

> 软架构视角：监控彭博社在 X（Twitter）上的推文，识别与中国 / AI / 半导体 / 光模块等相关内容，判断情绪，映射 A 股板块与龙头，并推送飞书。

---

## 1. 背景与目标

**目标**：持续监控彭博社在 X 上的官方账号推文，自动识别与**中国 / AI / 半导体 / 光模块**等相关的信息，判断**情绪**，映射到**A 股板块与个股（含龙头）**，并通过**飞书**推送结构化结果。

**非目标**（建议写进 PRD）：不做自动下单、不保证「因果预测」准确性、不替代合规与投研流程。

---

## 2. 关键约束（设计前必须对齐）

| 维度 | 说明 |
|------|------|
| **X/Twitter 数据获取** | 官方 API 有配额与费用；爬虫违反 ToS 且易被封。架构上应**优先 API**，并预留「人工/第三方数据供应商」通道。 |
| **账号范围** | 需明确：仅 `@Bloomberg`、`@business` 等列表；是否含转发/回复；是否多语言。 |
| **时效** | 推文到达 → 推送的 SLA（如 1～5 分钟）。 |
| **A 股映射** | 「龙头」需**可配置**（行业指数成分、市值、成交额、研报标签等），避免模型随口编造。 |
| **合规** | 日志脱敏、不外泄 API Key；若涉及付费数据，注意许可范围。 |

---

## 3. 总体架构（逻辑分层）

```
┌─────────────────────────────────────────────────────────────────┐
│                     Ingestion（采集层）                           │
│  X API / Webhook / 轮询 → 标准化 Tweet 事件 → 去重 → 入队          │
└────────────────────────────┬────────────────────────────────────┘
                             │
┌────────────────────────────▼────────────────────────────────────┐
│              Processing Pipeline（处理编排）                      │
│  过滤 → LLM 相关性+情绪+结构化抽取 → 板块/个股匹配 → 评分与裁剪      │
└────────────────────────────┬────────────────────────────────────┘
                             │
        ┌────────────────────┼────────────────────┐
        ▼                    ▼                    ▼
┌───────────────┐   ┌────────────────┐   ┌─────────────────┐
│  Knowledge    │   │  LLM Service   │   │  Notification   │
│  （板块/龙头） │   │  （推理+校验）  │   │  （飞书机器人）   │
└───────────────┘   └────────────────┘   └─────────────────┘
```

- **采集层**：只负责「拿到可靠、可审计的原始推文」。
- **编排层**：幂等、重试、死信队列、可回放。
- **知识层**：A 股代码、板块、龙头名单、别名（供匹配与约束模型输出）。
- **LLM 层**：相关性、情绪、实体与主题；**输出严格 JSON**，便于下游与飞书。
- **通知层**：飞书卡片/消息，带链接与摘要。

---

## 4. 模块设计

### 4.1 推文采集（功能 1）

**推荐实现路径**（按优先级）：

1. **X API v2**（若可用）：按 `user_id` 拉取时间线，或用 filtered stream（若账号在允许范围）。记录：`tweet_id`、`url`、`text`、`created_at`、`author`、`lang`、媒体/链接展开。
2. **轮询策略**：`since_id` 增量拉取；429 时指数退避；全局速率限制与 per-key 配额监控。
3. **去重**：以 `tweet_id` 做唯一键；处理「编辑推文」若 API 暴露则单独策略。

**数据模型（最小字段）**：

- `id`, `text`, `author_username`, `permalink`, `created_at`, `raw_json`（审计）

**输出**：写入消息队列（Kafka / Redis Stream / RabbitMQ / cloud 托管队列均可），事件名如 `tweet.received`。

---

### 4.2 相关性判断：用大模型而非纯规则（功能 2）

**结论**：**主路径用 LLM**，**辅以轻量规则与词典仅做预筛或后验校验**，原因：

- 规则难以覆盖「间接提及中国供应链、地缘政治、出口管制」等表述。
- LLM 适合多标签、弱监督、语义边界模糊的任务。

**推荐策略：两阶段**

| 阶段 | 作用 | 说明 |
|------|------|------|
| **Stage A：轻量预筛（可选）** | 降本 | 关键词/语言检测/长度；可省掉明显无关推文。**不要用复杂规则替代 LLM**。 |
| **Stage B：LLM 分类** | 主判断 | 多标签：`china_related`, `ai`, `semiconductor`, `optical_module`, `other_themes`；并给出 `confidence` 0～1。 |

**Prompt 设计要点**：

- 明确**标签定义与反例**（例如「光模块」与「光纤光缆」区分若业务需要）。
- 要求输出 **结构化 JSON**（键约定以仓库内 **`pipeline/prompts/triage_system.txt`** 为准；运行时归一化见编排层 `State.analysis`），禁止自由叙述为主输出。第 5 节 JSON 为含板块/个股映射的**终态目标示例**，可与当前 MVP 字段并行演进。
- **温度**：分类任务建议 `0～0.3`；需要稳定性可固定 seed（若供应商支持）。

**是否只用大模型**：**是（对「是否相关」这一层）**；**板块/代码**建议**检索 + 约束生成**（见 4.4），避免幻觉。

---

### 4.3 情绪判断（功能 3）

**定义**：对**标的资产或叙事**的情绪（利好/利空），而非对推文作者个人的情绪。

**实现**：

- 同一 LLM 调用中输出：`sentiment`: `positive` | `negative` | `neutral` | `mixed`，以及 `sentiment_confidence`。
- 可选：`sentiment_rationale` 一句（飞书里可折叠或仅内部日志）。

**注意**：短文本、反讽、仅标题党时 `neutral/mixed` 应合法；可设**置信度阈值**，低于阈值标为「不确定」并在飞书中提示。

---

### 4.4 A 股板块、个股、龙头（功能 4）

**原则**：**LLM 做「理解与归类」，代码级映射靠「知识库 + 检索」**。

**知识库内容（可版本化，如 Git + CI 或 DB）**：

- 申万/中信行业板块代码与名称（与产品对齐一种标准）。
- 股票：`code`, `name`, `aliases`, `industry`, `tags`（如 CPO、光模块、算力）。
- **龙头配置**：`sector_id` → `leader_stocks[]`（人工维护或规则：市值 TopN + 流动性 + 标签）。

**流水线**：

1. LLM 从推文抽取 **主题短语**（如「光模块」「GPU 出口限制」「台积电」）。
2. **向量检索 / 关键词检索**：在板块与股票库中召回 Top-K 候选。
3. **第二次 LLM 调用（可选但推荐）**：在**候选列表内**选择最终 `sectors[]`、`stocks[]`，并标注 `leader` 是否命中；若候选为空，则输出 `sectors: []` 并标记 `mapping_confidence: low`。

这样**龙头**来自配置或检索结果，而不是模型「随口点名」。

---

### 4.5 飞书推送（功能 5）

**触发**：仅当 `relevance` 命中业务定义的主题 **且**（可选）`confidence` / `mapping` 满足阈值。

**消息结构（建议用飞书交互卡片）**：

| 区块 | 内容 |
|------|------|
| 标题 | 来源 + 主题标签（如「彭博 · 半导体 · 偏空」） |
| 正文 | 推文摘要（可 LLM 压缩）+ **原始链接** |
| 元信息 | 时间、语言、情绪、置信度 |
| A 股影响 | 板块列表、个股列表、**龙头**高亮 |
| 原文 | `text` 截断展示 + 「展开」或链接跳转 X |

**幂等**：`tweet_id` 作为飞书消息去重键（飞书侧或自建 `sent_notifications` 表）。

**失败重试**：HTTP 5xx/429 重试；进入 DLQ 后告警（飞书/邮件）。

## 5. LLM 输出约定与 Schema 示例

**当前实现（MVP）**

- **系统提示**：`pipeline/prompts/triage_system.txt`（相关性、broad_push_eligible、嵌套 `analysis` 等）。
- **英译中系统提示**：`pipeline/prompts/translate_system.txt`。
- **图内字段**：模型返回经 `TweetTriageAnalyzer._normalize_payload` 写入 LangGraph **`PipelineState.analysis`**（见 `pipeline/state.py`），含 `is_relevant`、`broad_push_eligible`、`themes`、`keywords`、`sentiment` 等；**尚未**包含下述 `market_impact` 全量结构。

**终态目标示例**（便于工程落地与飞书渲染；含 A 股映射时可向此形状收敛）：

```json
{
  "tweet_id": "string",
  "relevance": {
    "china_related": true,
    "themes": ["semiconductor", "ai"],
    "confidence": 0.86
  },
  "sentiment": {
    "label": "negative",
    "confidence": 0.72,
    "rationale_short": "提及对华出口限制，偏供应链利空"
  },
  "market_impact": {
    "direction": "bearish",
    "sectors": [{"name": "半导体", "code": "SW801081"}],
    "stocks": [{"code": "603XXX.SH", "name": "示例", "role": "leader"}],
    "mapping_confidence": "medium"
  }
}
```

字段名可按内部规范调整，关键是**结构化 + 可追溯**。修改 triage 输出键时，请同步更新 **`triage_system.txt`** 与 **`_normalize_payload`**，并视情况调整本节约定。

## 6. 技术栈建议（可选组合）

- **运行时**：Python（生态适合 NLP 与异步）或 Go（高并发采集）+ Python worker。
- **队列**：云厂商托管队列或 Kafka。
- **存储**：PostgreSQL（推文、处理状态、通知记录）+ Redis（去重、速率限制）。
- **LLM**：OpenAI 兼容 API / 国内合规模型；**分类用小模型、复杂映射用大模型**可降本。
- **可观测**：OpenTelemetry + 结构化日志；关键指标：延迟、LLM token、推送成功率。

---

## 7. 安全与合规

- API Key 存密钥管理（KMS/Vault）；飞书 webhook 签名校验。
- 保留 `raw_json` 便于争议时审计；对外展示可做摘要脱敏。
- 明确**免责声明**在飞书模板 footer：自动分析，不构成投资建议。

---

## 8. 风险与缓解

| 风险 | 缓解 |
|------|------|
| X API 不可用或涨价 | 多数据源抽象；预算与降级策略（仅摘要人工频道） |
| LLM 幻觉标的 | 检索约束 + 白名单代码 |
| 延迟堆积 | 水平扩展 worker；热点账号单独队列 |
| 误报/漏报 | 人工反馈回路，修正标签与龙头库 |

---

## 9. 实施阶段建议

1. **MVP**：固定 1 个 Bloomberg 账号 + 队列 + 单次 LLM + 静态板块 JSON + 飞书文本消息。
2. **V1**：知识库检索 + 龙头配置 + 卡片模板 + 置信度门槛。
3. **V2**：反馈学习、多账号、多语言、A/B 不同 prompt。
